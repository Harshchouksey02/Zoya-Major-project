About Project2.1 Project TitleThe title of the project is “400°C FINE DINING RESTAURANT & IN-HOUSE TABLE BOOKING SYSTEM”.2.2 Project OverviewThe 400°C Fine Dining Restaurant & In-House Table Booking System is a full-stack web application designed to digitize restaurant operations and automate customer dine-in reservation workflows. The application replaces disjointed print-based menus, manual phone bookings, and complex third-party reservation widgets with a single centralized platform. Customers can access the responsive site to review menu offerings, dynamic chef details, testimonials, and book tables online. The in-house booking system validates inputs (limiting bookings from today to +1 month) and automatically assigns a table number. The system integrates simulated email and SMS receipt logging, while offering administrators a unified backend interface to perform CRUD on menu cards, chefs, testimonials, and manage reservations.2.3 The Problem StatementDine-in restaurants face several operational inefficiencies because of outdated workflows:Static paper menus are unhygienic, costly to print, and cannot easily reflect daily price changes or out-of-stock statusesIntegrating third-party reservation widgets leads to high monthly fees, loss of direct control over customer booking data, and lag in data syncing.Phone-based manual table bookings lead to frequent human errors, double-allocated tables and wait times.Customer inquiries and ratings are captured on paper feedback cards that are rarely digitized, making analysis impossible.Validation of booking parameters (like restricted future date ranges) is absent, allowing unmanageable bookings.2.4 The Proposed SolutionThe proposed solution is a unified, responsive web portal built using Python's Django framework and MySQLAn interactive menu card displaying dishes grouped by categories, complete with prices, ingredients, allergens, and spice rating scales.A custom In-House Booking Engine that handles table assignments, guest counts, and enforces date range boundaries (restricting reservations to 30 days in advance) on both the frontend and backend.Notification Simulation: The server generates booking receipts, printing email confirmation details and simulated SMS alerts to the command-line console logs.Secure Administration Dashboard: Replaces external setups, enabling managers to handle category uploads, add chefs, monitor reservation lists, and edit contents securely.2.5 Key Features SummaryStudent-side (Customer-facing) Features:Dynamic Home Screen: Beautiful banners, testimonials, dynamic stories, and active chef profiles.Menu Catalogue: Category tab filters (Starters, Mains, Desserts, etc.), allergens listing, and spice levels.Online Reservation Form: Captured inputs (Name, Phone, Email, Guest Count, Date via jQuery Calendar, Time).Time/Date Validation: Checks that bookings are not in the past and not beyond +1 month in the future.Contact Us Module: Validates and logs general customer inquiries with map links.TPO-side (Admin-facing) Features:Secure Panel Authentication: Password-protected access for database administrators.Menu CRUD Management: Full ability to create, read, update, and delete categories and menu items.Testimonial Control: Add/edit client reviews and display star ratings dynamically.Booking Monitor Board: Lists reservations, tracks guest volumes, and audits assigned tables.Static Site Control: Modify active homepage banners and active chef listings.2.6 Technology Stack — High-LevelThe application is built on a clean Python + MySQL architecture. The frontend uses HTML5, CSS3, JavaScript, and jQuery UI widgets for validation. The backend is powered by Python and Django framework, which manages MVT patterns, routing, security middlewares, and ORM database maps. The database layer runs on Standalone MySQL Server (Port 3306) managing relational tables.2.7 Scope of the ProjectThe project focuses on a single-restaurant outlet model managing local database configurations and digital reservations. The design is highly modular, meaning the database schema can support multi-outlet SaaS additions (via tenant ID fields) and payment gateways (e.g. Razorpay API integration) in subsequent iterations.3. Testing12.1 Testing StrategyA manual end-to-end testing approach was employed. Every form input, date check limit, and admin transaction was verified against defined test cases. Simulated SMS and email notifications were checked via server log streams.A comprehensive testing strategy was designed and executed to verify the functionality, reliability, and security of the "400°C Fine Dining Restaurant & In-House Table Booking System." Given the target audience (restaurant customers and administrative staff), the testing strategy combined structured manual validation with boundary testing to ensure a zero-defect deployment. The testing process followed a bottom-up approach, transitioning from individual units to complete integrated user flows.12.1.1 Testing MethodologyThe primary testing methodology adopted for this application is Black-Box Testing, focusing on validating the software against the defined functional requirements without requiring visibility into the internal codebase structures. The strategy was divided into four distinct phases:Unit / View Testing : Individual Django model methods, form validations (such as date and email constraints), and utility properties (such as the spice_range model helper) were tested in isolation to ensure they returned correct values.Integration Testing : Verified the communication between different components of the application. This included validating that the Django ORM successfully mapped classes to the MySQL database tables, that template tags dynamically rendered fetched database records (like categories and dishes), and that form submissions successfully updated database state.System & End-to-End (E2E) Testing: Simulating complete user journeys from both the customer and admin perspectives. For example, testing the complete cycle of a customer booking a table, verifying that the reservation is saved in the database, checking the table auto-allocation logic, and confirming that the admin can view, edit, or delete the booking from the Django Admin Panel.Boundary & Constraint Validation Testing: Testing the limits of inputs, specifically:Submitting historical dates in the booking form.Booking for dates beyond the 30-day booking window.Entering negative numbers or string values in guests and price inputs.Submitting empty mandatory fields to verify appropriate error display.12.2 Test EnvironmentTo ensure test results were consistent and represented real-world usage, the test suite was executed across a standardized local setup:Operating System : Windows 11 Home / ProfessionalBackend Server : Django Development Server (port 8000)Database Server : Standalone MySQL Server (v8.0)Client Browser : Google Chrome (v115+), Mozilla Firefox (v110+), and Microsoft EdgeTesting Tools : Browser Developer Consoles (Network & Console tabs) and Server Log Streams for monitoring SQL queries and print-statements.12.3 Test CasesThe full test register contains 40 cases.
